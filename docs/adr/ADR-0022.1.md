# ADR-0022.1: Policy Bypass Warning Events

**Status:** Accepted  
**Date:** 2026-05-23  
**Supersedes:** None  
**Related:** ADR-0022 (Typed RunEvent Unions), ADR-0023 (Enforcement Infrastructure)

## Context

Phase 23 introduced enforcement infrastructure with typed denial events (TRUST_DENIED, PAID_CALL_DENIED, etc.) that block operations when security policies are violated. However, not all code paths can be instrumented with enforcement gates:

1. **Unknown provider plugins** - Custom providers without enforcement hooks
2. **Custom HTTP clients** - User-provided HTTP clients that bypass instrumented wrappers
3. **Custom subprocess runners** - Direct subprocess calls that bypass enforcement
4. **Uninstrumented tools** - Tools that haven't been instrumented yet
5. **Upstream bypassed boundaries** - Enforcement bypassed by upstream code

When enforcement cannot be applied, the system needs to emit **non-blocking warnings** rather than blocking execution. This allows the system to continue operating while alerting users to potential security gaps.

## Decision

We introduce a new event type `POLICY_BYPASS_WARNING` that:

1. **Non-blocking** - Does not raise exceptions or block execution
2. **Rate-limited** - One warning per `(run_id, surface_identifier)` to prevent spam
3. **Actionable** - Includes `suggested_remediation` field with fix guidance
4. **Typed** - Uses discriminated union pattern from Phase 22

### Event Schema

```python
class PolicyBypassReason(StrEnum):
    UNKNOWN_PROVIDER_PLUGIN = "unknown_provider_plugin"
    CUSTOM_HTTP_CLIENT = "custom_http_client"
    CUSTOM_SUBPROCESS_RUNNER = "custom_subprocess_runner"
    UNINSTRUMENTED_TOOL = "uninstrumented_tool"
    UPSTREAM_BYPASSED_BOUNDARY = "upstream_bypassed_boundary"

class PolicyBypassWarning(BaseModel):
    schema_version: int = 2
    type: Literal["POLICY_BYPASS_WARNING"]
    timestamp: str
    run_id: str
    sequence: int
    data: PolicyBypassWarningData

class PolicyBypassWarningData(BaseModel):
    policy_id: str
    bypass_reason: PolicyBypassReason
    surface: str
    surface_identifier: str
    suggested_remediation: str
    parent_run_id: str | None = None
```

### Emission Helper

```python
def emit_policy_bypass_warning(
    run_id: str,
    sequence: int,
    policy_id: str,
    bypass_reason: PolicyBypassReason,
    surface: str,
    surface_identifier: str,
    suggested_remediation: str,
    parent_run_id: Optional[str] = None,
    emit_event: Optional[EventEmitter] = None,
) -> bool:
    """Emit non-blocking bypass warning with rate-limiting."""
```

### Rate-Limiting

Warnings are deduplicated per `(run_id, surface_identifier)` using a contextvar-based dedup set. This prevents warning spam when the same uninstrumented surface is called repeatedly within a run.

## Consequences

### Positive

- **Visibility** - Users are alerted to enforcement gaps without blocking execution
- **Actionable** - `suggested_remediation` field provides clear fix guidance
- **Non-disruptive** - Warnings don't break existing workflows
- **Auditable** - Warnings are logged in audit chains for compliance review
- **Scalable** - Rate-limiting prevents warning spam

### Negative

- **Not fail-closed** - Warnings don't prevent potentially unsafe operations
- **Requires instrumentation** - Adapter authors must emit warnings manually
- **Dedup complexity** - Rate-limiting adds contextvar state management

### Neutral

- **Adapter phases** - Phases 26-35 will emit bypass warnings when detecting uninstrumented code paths
- **UI integration** - Phase 22.1.5 adds UI banner for bypass warnings (deferred)
- **Query support** - `arc audit query --kind POLICY_BYPASS_WARNING` filters warnings

## Alternatives Considered

### 1. Block execution on uninstrumented code paths

**Rejected** - Too disruptive. Would break existing workflows that use custom providers or tools.

### 2. Silent logging without warnings

**Rejected** - Users wouldn't be aware of enforcement gaps. Warnings provide visibility and actionable guidance.

### 3. Per-session deduplication instead of per-run

**Rejected** - Per-run deduplication is more precise and prevents cross-run state leakage.
