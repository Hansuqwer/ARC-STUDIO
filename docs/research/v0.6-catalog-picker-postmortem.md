## Key findings

1. **Per-model capability filtering provides extreme precision.** The `/models --has vision` filter utilizes actual structured model records from `CostRates` rather than vendor-level coarse capability flags. This is validated by `test_has_vision_filter_per_model_granularity` where, e.g., `deepseek-v4-pro` has vision = False, whereas other vision models in the same vendor block show vision = True.

2. **Fail-closed capability lookup ensures robustness.** In `capability_gates.py`, requesting capabilities of an unknown model returns a dictionary with all features (`vision`, `tools`, `reasoning`) set to `False` instead of throwing an error or defaulting to permissive. This prevents UI lockups and satisfies the safety invariants of local-first capability gating.

3. **Offline catalog resolution prevents runtime latency.** By querying local `CostRates` records compiled in `openai_compatible.py` and `base.py`, commands like `/models` and `/model-info` operate completely offline, with 0 network overhead or dependency on external APIs (validated by mocking `urlopen` in `test_works_without_network`).

4. **Multi-vendor model ID overlap resolution.** The `get_capabilities` resolver successfully handles cases where model IDs overlap across vendors (e.g. `kimi-k2.6` appearing in both vendor-direct and `crofai` proxy blocks) by prioritizing the record containing non-empty `input_modalities` metadata.

5. **`ModelChanged` event diff semantics simplify downstream consumers.** By including `capabilities_added` and `capabilities_removed` list fields in the event envelope, consumers (like the TUI status bar and external telemetry) can instantly react to delta shifts without having to manually fetch and diff prior session states.

## Implications for next release train (v0.7-alpha & beyond)

- **Backfill Anthropic and OpenAI cost rows:** Currently, `/models --vendor anthropic` returns 0 results because Anthropic's model catalog resides in `AnthropicClient` rather than the `openai_compatible.py` `VENDOR_CONFIGS`. The next sprint should unify cost rows across all provider classes to enable full vendor coverage in search commands.
- **Support for `--max-context` filter:** Implement context window size metadata inside `CostRates` to power `/models --max-context <tokens>` searches.
- **Opt-in runtime catalog feed:** To support dynamic pricing and model catalog updates without shipping new Python versions, design the signed v0.7-alpha runtime feed while maintaining local-first fallbacks.
