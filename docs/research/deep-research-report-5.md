# ARC Studio MCP roadmap

## MCP opportunity map

From the baseline you provided, ARC already has the beginnings of a credible MCP server: local stdio transport, a real tool/resource surface, workspace trust re-checks, output caps/redaction, and audit events. The gap is that the MCP ecosystem has moved decisively beyond “local stdio helper server” into remote, authenticated, multi-server, host-managed workflows. The latest stable MCP specification is version **2025-11-25**. It standardises server primitives such as **tools, resources, and prompts**; client features such as **sampling, roots, and elicitation**; **Streamable HTTP** for remote transport; **OAuth-based authorisation** for HTTP transports; and **experimental tasks** for durable, asynchronously retrieved request state. The March 2025 spec revision also formally replaced legacy HTTP+SSE as the preferred remote transport with Streamable HTTP. The current draft is already exploring a more stateless model with per-request capability declaration, so ARC should target the stable spec now while avoiding public APIs that depend too heavily on today’s stateful assumptions. citeturn2view0turn4view0turn6view0turn6view1turn8search4turn21search10

That shift is visible in product ecosystems. Claude now supports remote MCP across hosted products, a reviewed Connectors Directory, MCP Apps, and Claude Desktop extensions packaged via MCPB. Cursor supports stdio, SSE, and Streamable HTTP; OAuth and static OAuth configuration; first-class **tools, prompts, resources, roots, elicitation**, and **Apps**; plus marketplace/install-link workflows, permissions, and enterprise registration APIs. Codex supports stdio and Streamable HTTP in both CLI and IDE contexts, includes bearer/OAuth support, reads server `instructions`, and exposes approval gates for side-effecting MCP tool calls. Continue supports MCP in agent mode, remote transports, secrets handling, and integration cookbooks, though its public docs emphasise tools and context-provider usage more strongly than a rich first-class surface for prompts/resources. citeturn35view0turn24view0turn24view1turn15view0turn24view2turn33search12turn16search0turn30search0turn31search7

The external server landscape is also now good enough to justify ARC making the jump. Official and de facto reference servers already cover developer-critical categories: **filesystem**, **git**, **memory**, **fetch**, **browser automation via Playwright**, **GitHub**, **Arize Phoenix**, and multiple database patterns including PostgreSQL and SQL-focused MCP servers. This means ARC does not need to guess where demand is; it can prioritise the categories that are already proving valuable in Claude, Cursor, Codex, Continue, GitHub Copilot, and developer workflows more broadly. citeturn19search0turn19search2turn25view0turn17search0turn26search18turn20search1turn20search16

The most important implication is that ARC’s opportunity splits cleanly into three bands. **Parity** means adding Streamable HTTP, HTTP auth, prompts/resources beyond the current minimal baseline, and host-side client features such as roots and elicitation. **Ecosystem fit** means becoming an MCP **host/hub** as well as a server: registry discovery, external server management, permissions, task/progress UX, and replayable auditing. **Differentiation** means exposing ARC-native capabilities that generic MCP hosts do not have, especially **SwarmGraph wrappers**, **memory graph resources**, and observability-aware resources that blend ARC’s own state with servers like GitHub and Phoenix. citeturn23view0turn4view6turn25view3turn27view0turn17search8

A final strategic point: the official MCP Registry is important, but it is still in preview and its own documentation explicitly says it is intended primarily for downstream aggregators and compatible registries, not direct consumption by host applications. GitHub’s enterprise MCP management, meanwhile, already assumes allowlisted registries and policy enforcement. ARC should therefore support **curated and private registries** first, with the official registry treated as upstream metadata rather than the only discovery source. citeturn4view6turn5view0turn25view3

## Security model for MCP HTTP

ARC should add **Streamable HTTP** as the default remote transport and keep stdio for local subprocess workflows. Under the stable spec, a Streamable HTTP server exposes a **single MCP endpoint** that accepts POST and GET, may use SSE to stream multiple server messages, and may assign a secure `MCP-Session-Id` during initialisation. Clients then send `MCP-Protocol-Version` on subsequent requests, and sessions can be explicitly torn down with HTTP DELETE. The spec also warns that local HTTP servers must validate `Origin`, should bind to localhost rather than `0.0.0.0`, and should require authentication. In practice, this is the minimum contract ARC needs if it wants to interoperate with modern remote clients and servers rather than stay trapped in stdio-only workflows. citeturn6view0turn4view0

For HTTP authentication, ARC should implement the spec’s **OAuth 2.1-based** approach, but with a compatibility layer that reflects the real host ecosystem. The stable spec requires protected resource metadata for authorisation server discovery, requires clients to support metadata/OIDC discovery, prefers **Client ID Metadata Documents**, allows **Dynamic Client Registration** as fallback, requires **PKCE S256**, and forbids access tokens in URI query strings. It also explicitly says the HTTP auth spec should **not** be used for stdio, where credentials should instead come from the environment. Cursor already supports both dynamic OAuth and static client credentials in config; Claude supports DCR, Client ID Metadata Documents, Anthropic-held credentials, and custom credential flows; Codex supports bearer token and OAuth authentication. ARC should therefore support three modes on day one: spec-native OAuth discovery, static enterprise client credentials, and environment-based stdio credentials. citeturn36view0turn36view1turn36view2turn36view4turn24view0turn15view0turn24view2

The recommended **local auth** model for ARC HTTP is: loopback-only binding by default; a short-lived bearer token minted by the ARC launcher; storage in process memory or the OS keychain; explicit `Origin` validation even on localhost; and no reuse of `MCP-Session-Id` as an authentication secret. That is not a verbatim spec requirement, but it follows directly from the transport security guidance, the ban on query-string tokens, and the broader OAuth/MCP security recommendations. ARC should also treat browser callback URLs as high-risk: exact redirect URI matching, no wildcard redirects, and visible callback hostnames in all approval UIs. citeturn6view0turn36view3turn36view4turn37view0

ARC’s permission model should be split into four layers rather than one coarse “trusted/untrusted” bit. **Connection trust** decides whether a server may connect. **Root trust** decides which workspace/file roots it may see. **Capability trust** decides whether a server may expose tools, prompts, resources, sampling, or elicitation at all. **Operation approval** decides whether a particular call may execute. This is consistent with the MCP spec’s insistence on explicit user consent for data access and tool invocation, with Claude’s tool hints and connector permissions, with Cursor’s default tool-confirmation flow and permissions file, and with Codex’s explicit treatment of destructive MCP actions as approval-worthy events. ARC should map `readOnlyHint`, destructive annotations, network/open-world behaviour, and workspace scope into first-class UI policy. citeturn2view0turn35view1turn35view2turn15view0turn33search12

ARC also needs to fully implement **elicitation and sampling safety** if it becomes an MCP host. Elicitation form mode is for non-sensitive structured input only; secrets, tokens, passwords, payment details, and OAuth-like interactions must use **URL mode**. Sampling requests should always have a human-in-the-loop approval surface, with prompt review, editability, and result review before the server receives the outcome. This is especially important if ARC later adds SwarmGraph wrappers or recursive agent behaviours, because the protocol deliberately gives the client—not the server—control over model access and sensitive user interactions. citeturn6view4turn7view0turn2view0

Finally, ARC’s HTTP security model should explicitly defend against the MCP-specific attacks that the official security guidance calls out: **confused deputy**, **token passthrough**, **SSRF**, **session hijacking**, over-broad scopes, and unsafe one-click local server installation. Concretely, that means: no token passthrough to downstream APIs; per-client consent for proxy-style authorisation flows; exact redirect URI validation; SSRF-safe discovery that blocks internal IPs, cloud metadata endpoints, and non-loopback HTTP except in explicit dev mode; least-privilege scopes; sandboxed local server execution; and a consent dialog that shows the full command before starting any local server. ARC already has audit events and output caps; it should extend that foundation into **replayable**, redacted, immutable traces that record approvals, roots, scopes, arguments, and result digests, with replays defaulting to dry-run unless a user explicitly approves side effects again. citeturn37view0turn21search12turn15view0turn33search12

## MCP client and hub architecture

The core design rule should be: **follow the MCP host/client/server architecture literally**. The host application creates **one client per connected server** and manages security, consent, context aggregation, and orchestration. ARC should not collapse everything into one giant synthetic “uber-server” by default. That keeps trust boundaries clearer, matches the spec, and makes it easier to reason about per-server policies, crash isolation, and auditing. citeturn8search3turn22search5

A good ARC hub therefore needs six internal subsystems. First, a **catalogue/discovery layer** that knows about ARC’s own servers, curated registries, and project-local definitions. Second, a **connection manager** that can spawn stdio servers, connect to HTTP servers, handle session IDs, and refresh auth. Third, a **policy engine** that applies org policy, project trust, roots, and tool permissions. Fourth, a **context router** that decides which tools/resources/prompts to expose to the model in a given turn. Fifth, a **task manager** for long-running operations, retries, cancellation, and progress. Sixth, an **audit store** that records every state transition. The official MCP client best-practices doc is especially relevant here: once a host accumulates many servers and hundreds of tools, the naïve “load everything into the prompt” approach breaks down badly. ARC should adopt **progressive tool discovery** from the start, with a lightweight tool-search/search-capability layer instead of injecting full tool definitions all the time. citeturn23view0

ARC should support two user-facing modes. In **server mode**, ARC continues to expose its own primitives to other MCP hosts, but now over both stdio and Streamable HTTP. In **hub mode**, ARC becomes a first-class MCP host that can connect to local and remote servers, manage auth, and present a unified UX over external tools/resources/prompts. For enterprise or self-hosted deployments, path-based mounting of multiple servers under one HTTP app is a sensible operational pattern; the official Python SDK explicitly demonstrates mounting multiple Streamable HTTP servers under different paths, and the TypeScript SDK examples discuss distributed message routing for cases where session state lives on one node but termination happens elsewhere. That gives ARC a clean way to build an “ARC hub service” later without inventing a proprietary composition model. citeturn28search7turn29search7

External server management should be opinionated. ARC should support version pinning, lockfiles, per-project enablement, install records, server crash diagnostics, retry/backoff, health checks, and dynamic refresh of tool/resource/prompt lists. Cursor already exposes programmatic registration, per-project/global configs, toggles, logs, and auto-run permissions. Claude Code already exposes a status panel, dynamic `list_changed` refresh, and HTTP reconnection behaviour. ARC should match those basics, then go further with richer provenance and replay. citeturn15view0turn24view3

On registries, ARC should not wire itself directly to the official public registry as if it were the only source of truth. The official registry documentation explicitly pushes host applications toward downstream, OpenAPI-compatible registries and notes that private registries are a valid pattern; GitHub’s enterprise MCP management adds exactly the kind of registry allowlisting and policy enforcement that larger customers will expect. The right ARC move is to support: a public curated ARC registry, private enterprise registries, and import/synchronisation from upstream public sources. citeturn4view6turn25view3

The advanced protocol features should slot naturally into this architecture. **Roots** map to ARC’s trusted workspace model. **Prompts** become slash commands and workflow starters. **Resources** become attachable artefacts and side panels. **Elicitation** becomes structured modal/UI capture. **Sampling** becomes a reviewed sub-agent call surface. **Tasks** become long-running, observable operations with progress, cancellation, and deferred result retrieval. That architecture gives ARC parity with the spec while leaving room for its own differentiators. citeturn6view2turn6view3turn6view4turn7view0turn8search10

## IDE integration ideas

ARC should add a dedicated **MCP panel** rather than burying MCP behind settings and logs. The panel should show connected servers, transport, auth status, trust state, roots, available tools/resources/prompts, permissions, task activity, and failures. Cursor’s MCP UX already demonstrates the value of project/global config, easy enable/disable, approvals, logs, and install flows; Claude Code’s `/mcp` panel shows the value of an at-a-glance operational view. ARC can combine those ideas into a more IDE-native inspector. citeturn15view0turn24view3

ARC should also treat **prompts and resources as first-class IDE objects**. Prompts belong in slash-command pickers, command palettes, and “workflow starters”. Resources belong in an attach menu, explorer pane, and diffable side view. The spec explicitly positions prompts as user-controlled templates and resources as client-managed context attachments, and Claude already surfaces remote resources/prompts through the attachment flow. ARC should bring the same concept into code workflows: attach a workspace index resource, a memory-graph resource, or a failing-CI prompt without writing a wordy manual instruction each time. citeturn6view2turn8search7turn23view1

Tool permissions deserve their own **policy centre**. Users and admins should be able to allow, ask, or deny by server, tool, capability class, workspace, domain, or environment. ARC should make destructive/network/open-world tools highly visible, show exact arguments before approval, and remember approvals at the right scope. That is consistent with MCP’s security principles, Claude’s connector/tool permission model, Cursor’s approval/auto-run model, and Codex’s reviewer model for side-effecting tools. citeturn2view0turn35view2turn15view0turn33search12

Long-running work should appear as **task timelines**, not opaque spinner states. If ARC adds experimental MCP tasks and transport streaming, users should see creation time, progress messages, nested tool calls, approvals, retries, and final outputs, with the ability to cancel, retry, or attach the resulting resource into chat. That would make SwarmGraph wrappers feel native rather than bolted on. citeturn8search10turn6view0

Finally, ARC should consider an **Apps-ready** view model even if it does not ship full MCP Apps support in phase one. Claude and Cursor already support MCP Apps, and the MCP Apps extension provides a path for sandboxed interactive panes built on standard web primitives. Even if ARC starts with simple HTML/markdown preview panes for resources, designing the panel system with an eventual “interactive MCP view” model will save a lot of refactoring later. citeturn23view2turn15view0turn35view0

The acceptance bar should be concrete and testable. ARC should not consider the work “done” until it passes at least the following cross-cutting checks. First, **wire compatibility**: stdio and Streamable HTTP both work with the MCP Inspector and with a representative set of public servers, including filesystem, Playwright, GitHub, Phoenix, and memory. Second, **auth correctness**: OAuth discovery, PKCE, static-enterprise credentials, and env-based stdio secrets all work, and known-bad patterns—query tokens, missing PKCE metadata, unsafe HTTP discovery, token passthrough—are rejected. Third, **policy correctness**: roots actually gate file access, destructive tools require approval, URL-mode elicitation is enforced for sensitive flows, and tool approvals can be audited. Fourth, **user-journey quality**: servers can be installed, authorised, enabled/disabled, debugged, and removed entirely from the IDE panel without hand-editing config files. Fifth, **task visibility**: long-running operations show progress, survive refreshes where appropriate, and can be cancelled or replayed safely. citeturn19search1turn19search2turn25view0turn17search0turn27view0turn36view0turn36view1turn37view0turn15view0turn24view3

If ARC wants a simple release gate for “phase one is good enough”, use this one: **a user can connect ARC to a remote MCP server over Streamable HTTP, authenticate safely, attach a resource, run a prompt, approve a tool call, observe a long-running task, and inspect an audited replay trail—all without leaving the IDE**. ## Open questions and limitations

The largest uncertainty in this research is not the MCP spec itself, but the **publicly documented host support** in some clients. Continue’s public docs clearly document MCP in agent mode, remote transports, and context/tool integration, but they do not clearly document first-class MCP prompt/resource surfaces or explicit support for roots, elicitation, sampling, or Apps in the sources reviewed here. That is why the compatibility matrix marks those areas as partial or undocumented rather than assuming parity. citeturn16search0turn31search7turn30search0

Codex’s public docs clearly document stdio/HTTP support, bearer/OAuth auth, shared config across CLI and IDE, server instructions, and approval handling for side-effecting MCP tool calls. They also discuss MCP conceptually in terms of tools/resources/prompts. But the specific public host docs reviewed here do not clearly document explicit support for roots, elicitation, sampling, or MCP Apps, so those were also marked as partial/undocumented rather than absent. citeturn24view2turn33search4turn33search12
