## Key corrections from the original research doc

1. **Ed25519 → hash-pinning.** The arena.ai research doc proposed Ed25519
   signature verification against `api.arc-studio.ai`. That domain doesn't
   exist and neither OpenRouter nor models.dev sign their feeds. The v3 spec's
   hash-pinning approach is correct: any change from the pinned SHA-256 requires
   explicit `arc pricing-feed accept-new-hash`. Stricter in practice (every
   change reviewed) though weaker in cryptographic provenance.

2. **Remote telemetry broker → local-sanitized only.** The research doc's
   "Shared OTel Broker" sent run data to a remote endpoint. That violates
   local-first.md. The shipped observability bridge: per-session consent +
   `sanitize_attributes()` strips prompt/code/context, exporting only semantic
   counts (tokens, cost, durations) to a user-chosen destination.

3. **ModelChanged wiring uses the internal bus**, not Textual messages.

## What's a stub (honest accounting)

- `arc pricing-feed accept-new-hash` — the client method exists and logs
  acceptance, but the CLI subcommand group and the persistent hash store
  (`~/.arc/pricing_feed.toml`) are NOT wired. A follow-up sprint (candidate D
  in post-v0.7-options) should land the CLI + persistence.
- Observability bridge `_send()` — sanitization + consent are real and tested;
  the actual OTLP wire-up is environment-specific and only activates when
  `opentelemetry-exporter-otlp` (optional extra) is installed.
