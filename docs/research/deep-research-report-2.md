# ARC Studio as a better AI IDE

## Strategic diagnosis

ARC Studio should not try to win by becoming “yet another chat sidebar”. The current market leaders have already raised the baseline: Cursor combines indexed codebase search, checkpoints, queued prompts, browser tooling, cloud/web/mobile agent surfaces, PR review via Bugbot, and enterprise governance features such as hooks, rules, audit logs and sandboxing; Windsurf couples checkpoints with preview-driven debugging, reusable workflows, hooks, terminal auto-execution controls and multi-model arena runs; and VS Code + Copilot now spans editor-first and agent-first workflows with reviewable edits, checkpoints, memory, hooks, MCP, plugins, background/cloud agents, and built-in trust controls like tools pickers and network filters. citeturn30view0turn6view2turn7view0turn34view0turn34view1turn34view2turn34view3turn34view4turn26search3turn26search4turn26search6turn26search8turn26search15

The opening for ARC is different. Based on the surface you listed, ARC already has the ingredients most competitors do not have in one place: traces, audit chains, HITL/audit UI, SwarmGraph topology/consensus/cost views, sandbox policy, provider configuration, live/replay/disconnected states, a session bridge, and a Battle tab. That means ARC can plausibly become the **best trace-native agent engineering cockpit** rather than merely a better autocomplete tool.

The products that most clearly point to ARC’s opportunity are OpenHands, which is built around an append-only event log and stateless step loop, Zed, which is strong at parallel agent orchestration and worktree isolation, JetBrains/Junie, which has polished rollback and review behaviour, and terminal-first systems like Claude Code and Codex CLI, which make permissions and checkpoints first-class. None of them, however, combine **IDE-native coding, replay-grade observability, graph-native multi-agent visualisation, and enterprise auditability** in one coherent product. That is the space ARC can own. citeturn32view1turn32view2turn38view0turn37view0turn29view1turn29view2turn28view1turn28view3

## Prioritised ARC IDE backlog

The order below puts **market-parity blockers** first, then moves into **ARC-native differentiation**. The common pattern across Cursor, VS Code, Windsurf, Claude Code, Codex, JetBrains, Continue and Cline is that the winning AI IDEs now expose clear loops for **understand → plan → apply → review → verify → resume**, with explicit controls over tools, permissions and rollback. citeturn30view0turn26search7turn34view0turn29view1turn28view0turn37view1turn39search1turn35view1

| Priority | Backlog item | Why it belongs here |
|---|---|---|
| **P0** | **Trace-aware Review mode** | Without this, ARC cannot convert traces/audit chains into user trust. |
| **P0** | **Workspace Intelligence layer** | Every leading tool now has a strong codebase-understanding story; ARC must have one. |
| **P0** | **Explicit Plan / Apply / Review loop** | Needed for predictable multi-file work and safer HITL review. |
| **P0** | **Structured Test Bench** | Verification must live next to agent output, not in a separate terminal tab. |
| **P0** | **Approval Centre and policy engine** | Essential because ARC already has sandbox policy and HITL/audit ambitions. |
| **P0** | **Replay debugger for failed agent runs** | This is where ARC’s trace-native edge becomes tangible. |
| **P1** | **Inline actions + edit basket** | Improves day-to-day coding flow and reduces chat-only dependence. |
| **P1** | **Terminal cockpit** | Makes terminal context, command approvals and verification much more fluent. |
| **P1** | **Onboarding and provider/MCP setup wizard** | Directly improves activation and reduces failed first-run experiences. |
| **P1** | **Notifications + session bridge continuation** | Necessary once runs become longer-lived and multi-surface. |
| **P2** | **SwarmGraph Commander** | High differentiation, but depends on good replay and approval primitives underneath. |
| **P2** | **Battle tab upgrade into model/agent compare mode** | Valuable, but only after review and trace inspection are strong enough. |
| **P2** | **Assurance dossiers and exportable audit packs** | Excellent enterprise differentiator once evidence pipelines are stable. |
| **P2** | **Responsive/mobile shell** | Important, but should ride on top of session-bridge and notification work. |
| **P2** | **Org analytics on run quality, cost and approval friction** | Powerful for management and platform teams, but not a first-run user blocker. |

The **must-have for v1** set should therefore be: **Trace-aware Review mode, Workspace Intelligence, Plan/Apply/Review, Test Bench, Approval Centre, Replay Debugger, Terminal Cockpit, and Guided Onboarding**. That group covers the minimum credible parity for a modern AI IDE while also unlocking ARC’s distinctive assets.

The **differentiator set** should be: **SwarmGraph Commander, Battle tab as evidence-backed model compare, assurance dossiers, capability provenance for local MCP, replay-linked diffs, and policy explainability**. These are the features that competitors only approximate in fragments. Cursor and VS Code have governance; Windsurf and Zed have multi-agent/model UX; OpenHands has event architecture; Claude/Codex have disciplined approvals. ARC can combine all of those in one coherent surface. citeturn7view0turn26search6turn34view4turn38view0turn32view1turn29view1turn28view1

## UX wireframes and acceptance criteria

**Command centre.** The default desktop shell should have a left rail for **Sessions / Runs / Workflows / Battle / Assurance**, a centre panel that can switch between **Plan / Apply / Review / Replay**, and a right inspector for **Evidence / Approvals / Tests / Costs**. The purpose is to bring ARC closer to VS Code’s Agents window and Zed’s thread sidebar, but with much richer operational context. citeturn26search3turn38view0

**Review mode.** The centre shows a grouped diff tree by task and file. The right panel shows “why this changed”, “which tests passed”, “which tool or MCP resource informed it”, “which approval unlocked it”, and “what policy applied”. A sticky footer offers **Accept selected**, **Reject selected**, **Request fix**, and **Rewind**. This is the single most important new surface for trust. It should feel closer to JetBrains’ rollback workflow plus VS Code’s review-edits than to chat transcripts. citeturn37view0turn37view1turn26search14

**Replay debugger.** The top has a scrubber with step markers for prompt, search, tool call, edit, test, approval, failure and checkpoint. The left shows the step list; the centre shows code/output at that step; the right shows structured metadata: tokens, cost, elapsed time, tool name, MCP origin, approval status, and related SwarmGraph node. OpenHands’ event-log architecture is the closest precedent, but ARC should make it much more inspectable. citeturn32view1turn32view2

**SwarmGraph Commander.** The graph canvas lives in the centre; the right side contains **topology**, **consensus**, **cost**, and **risk** panels; the bottom contains a “node compare” tray. Clicking a node opens its transcript, code delta, tests, approvals and replay anchor. Windsurf’s arena and Zed’s parallel-agents UX suggest the interaction model; ARC’s edge is that the graph is not just visual, but evidence-linked. citeturn34view4turn38view0

**Approval centre.** This should be a dedicated queue, not a transient toast. Each card should show: command/tool, risk, scope, policy reason, sandbox mode, projected side effects, and alternative safe path. The user can approve once, for this run, for this workspace or via a policy rule. Claude Code, Codex, Cursor, Cline and VS Code all point in this direction; ARC should unify those ideas in one screen. citeturn29view1turn28view1turn7view0turn35view2turn26search2

**Compact and mobile triage.** On smaller screens, ARC should not attempt full coding. It should prioritise **status cards** for live/replay/disconnected sessions, pending approvals, failing tests, run summaries and accept/reject decisions, with one-tap resume on desktop. Cursor’s web/mobile agents, VS Code’s browser Agents window and Claude Code’s web/mobile surfaces make this a reasonable expectation for serious agent workflows. citeturn6view2turn26search3turn29view0turn29view1

The top-ten features should ship with explicit acceptance criteria like these:

| Feature | Acceptance criteria |
|---|---|
| **Trace-aware Review mode** | Every diff hunk can show at least one originating step or explicitly say “manual/unknown”; users can accept/reject at hunk, file and task levels; rejecting a hunk does not discard unrelated accepted edits; audit export includes prompts, tools, approvals and test evidence. |
| **Workspace Intelligence layer** | Initial indexing completes for a medium monorepo without blocking the UI; incremental updates land after edits; search results include files, symbols, traces and MCP resources in one list; each result explains why it matched. |

| **Approval Centre** | All risky tools and commands enter one queue; policy rationale is visible before approval; users can approve once/session/workspace/project; every approval and rejection is immutable in the audit chain. |
| **Replay debugger** | Users can scrub any completed run step-by-step; restoring a checkpoint reconstructs code state accurately; comparing two runs shows changed plan, changed tool use and changed diffs; failed steps are filterable. |

## Open questions and limitations

I did not have access to ARC Studio’s private repository or internal connector documentation in this session, so the Theia module suggestions above are **recommended extension boundaries**, not verified file paths.

The most important unresolved product questions are architectural rather than conceptual. First, where ARC’s trace schema currently lives, and whether it already captures step-level file diffs, tool I/O, approvals and test results in a reusable way. Second, whether SwarmGraph today is purely observational or already participates in orchestration decisions. Third, whether the browser-only deployment model imposes performance constraints for large-repo indexing, step replay and heavy graph rendering. Those answers will affect implementation sequencing, but they do not materially change the strategic conclusion: **ARC should become the best trace-native AI IDE for agent engineering, not a better generic chat client.**
