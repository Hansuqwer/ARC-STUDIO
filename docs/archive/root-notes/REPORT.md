What's **actually missing** at HEAD is not new exotic features. It is:

1. **Wiring**: a lot of these subsystems are independently implemented but not joined to the IDE, the daemon SSE bus, or each other.
2. **MCP risk *enforcement***: `mcp/manifests.py` *classifies* tool risk and detects drift, but no MCP-call-time enforcement consumes that risk on outbound MCP client calls.
3. **Streaming verifier in IDE/daemon**: CLI works; arc-extension still trusts non-streamed `verify` endpoints in the Assurance tab.
4. **Adapter coverage of HMAC**: per the roadmap "ARC does not claim adapter-wide keyed audit coverage."
5. **AGENTS.md / Skills surface**: the repo *has* an `AGENTS.md` for its own development, but ARC Studio does not yet *consume* `AGENTS.md` from a user workspace as a first-class agent-context primitive.
6. **A2A**: zero adoption, despite Linux Foundation governance and 150+ orgs in production [A2A].
7. **MCP `2026-07-28` stateless RC posture**: ARC has not yet positioned for the upcoming spec change [MCP-RC].

### Top 5 ecosystem shifts (2026)

1. **MCP "won" the agent-to-tool layer.** Anthropic, OpenAI, Google, Microsoft, AWS all ship native MCP; 78% of enterprise AI teams report at least one MCP-backed agent in production [Pickaxe-A2A-MCP]. Codex's own monorepo ships 88 nested AGENTS.md files. Research across 138 real repos: developer-authored AGENTS.md improves agent success ~4% and reduces agent-generated bugs 35-55% [DanielVaughan-AGENTS].
4. **Durable, checkpointed, time-travelable execution is the new baseline.** LangGraph 1.2 (May 11, 2026) ships `interrupt()` / `Command(resume=...)` with `AsyncPostgresSaver` checkpointers; pause/resume + horizontal scaling are first-class [LangGraph-Durable][LangGraph-Tutorial]. AutoGen folded into **Microsoft Agent Framework 1.0 GA (April 3, 2026)**, also durable + native MCP+A2A [VSM-MAF][DigitalApplied-MAF]. CrewAI 2026 added checkpoint forking with lineage tracking, Flow state export, reasoning-token tracking [CrewAI-Changelog].
5. **OpenTelemetry GenAI semconv + OpenInference are the de-facto observability fabric.** `genai.*` attributes, `openinference-instrumentation-mcp` for cross-MCP context propagation, AgentCore/Phoenix/Datadog all standardize on it [Uptrace-OTel][Arize-Phoenix][Augment-Obs]. **A2A** (Google → Linux Foundation, v1.2 March 2026) handles agent-to-agent; **complementary to MCP, not competing** [Pickaxe-A2A-MCP][StellaAgent-A2A][TheNextWeb-A2A].

### Top 5 repo-verified ARC/SwarmGraph gaps (after Repo Reality Pass)

Distinct from the "feature list" gaps — these are real:

1. **MCP-client risk enforcement.** `mcp/manifests.py` *scores* tool risk; `arc mcp workbench` *inspects* manifests; but `arc mcp serve` only enforces trust on *server-side* tools that ARC itself exposes. There is no per-call gate on **outbound** MCP tool calls made by adapters or by an inner Agents-SDK/LangGraph agent that ARC supervises. **True gap.**
2. **A2A + AGENTS.md adoption.** Neither is present in code (`grep -r "a2a\|agent-card" python/src` = 0 hits relevant to A2A protocol; `AGENTS.md` exists only as ARC's own engineering rules). **True gap, low cost.**
3. **Streaming verifier wired only to CLI, not Assurance tab / daemon SSE.** `packages/arc-extension/src/browser/**` has audit-verify UI but consumes the legacy daemon route, not the streaming verifier's stable JSON contract (`{ok, mode, records_checked, reason, duration_ms}`). **Hardening.**
4. **Eval-to-Policy → actually wired into runs.** `evals/policy_recommend.py` produces `PolicyRecommendation` objects but no automatic "apply on next run" or saved profile loop closes back into `security/policy_linter.py` or runtime profile selection. **Partial → hardening.**
5. **Capability Cards are produced but no consumer enforces them.** `capabilities/from_adapters.py`, `from_ir.py`, `from_mcp.py` build cards; `capabilities/signing.py` signs them; but no runtime call site reads a signed card and *refuses* execution when `audit_level < required` or when `signature_invalid`. The infrastructure exists; the gate does not. **Partial → hardening, single highest-leverage missing wire.**

Other gaps that are real but smaller: OTel/OpenInference exporter for `RunEvent`s (`telemetry/` exists but I did not verify GenAI semconv coverage end-to-end); A2A AgentCard generator at `/.well-known/agent-card.json`; MCP `2026-07-28` `_meta` posture; signed AgentCard verification.

### Top 3 build-now recommendations

1. **Capability Card Enforcement Gate (F1)** — Already-built cards become a *runtime gate*. Wire `capabilities/registry.py` lookups + signature verification into adapter `run_workflow()`, `mcp/server.py` tool dispatch, and `swarmgraph.runner`. **MVP: 2 days. Largest impact per dev-hour because everything upstream is already done.**
2. **MCP Tool Risk Sandbox v2 — Outbound Enforcement (F2)** — Extend `mcp/manifests.py` risk classifier into an outbound-call gate consumed by ARC's MCP *client* code paths and by adapter wrappers around third-party MCP-client SDKs. Add a deterministic + LLM-free re-scorer using `security/injection_patterns.py`. **MVP: 3-4 days.**
3. **AGENTS.md Workspace Ingestion + Skills Surface (F3)** — Make ARC the first agent IDE that *audits, signs, and pins* an `AGENTS.md` (and nested files) as part of the trust state, and exposes them to runtime adapters via `protocol/`. Pairs with capability cards. **MVP: 2-3 days.** Bonus: lets ARC interoperate with Codex/Cursor/Copilot/Claude Code without reinventing config.

## 3. Source-Backed Ecosystem Research

### 3.1 MCP — protocol direction and security

- **Spec direction (2026-07-28 RC).** Stateless envelope (`_meta` carries protocol version, client identity, negotiated capabilities), new `server/discover` method, multi-round-trip pattern for server-initiated calls (SEP-2322), OAuth/OIDC tightening (DCR `application_type` SEP-837, issuer-bound credentials SEP-2352, scope accumulation on step-up SEP-2350, well-known suffix SEP-2351, mix-up defense SEP-2468) [MCP-RC: https://mcp.directory/blog/mcp-2026-07-28-release-candidate].
- **Transports.** stdio is "recommended for local… securely sandboxed by OS, ideal for local tools" — exactly ARC's posture. Streamable-HTTP is becoming default for *remote* and is what Claude Code's reference docs flag as recommended remote default as of 2026-05 [NetworkIntelligence-MCP; Hidekazu-CC-2026].
- **Security baseline.** CoSAI 2026 whitepaper: sandboxing/isolation **required** for MCP servers and any execution of LLM-generated code; containers alone are insufficient; HITL on sensitive actions; users cannot disable confirmations on privileged tools; signed artifacts + SBOM + TLS + end-to-end signatures [Adversa-CoSAI: https://adversa.ai/blog/mcp-security-whitepaper-2026-cosai-top-insights/]. Practical-DevSecOps: tool registry should be **digitally signed, version-locked**, vetted; dynamic policy enforcement using full call context (prompt + user + tool + params + target); incident-response with rapid tool rollback [Practical-DevSecOps: https://www.practical-devsecops.com/mcp-security-vulnerabilities/]. SOC Prime: "split read and write tools, review scopes regularly, keep scope sets minimal and task-specific" [SOCPrime-MCP: https://socprime.com/blog/mcp-security-risks-and-mitigations/]. NetworkIntelligence: **43% of public servers vulnerable to command injection; 4 critical CVEs 7.3-9.6** [NetworkIntelligence-MCP: https://www.networkintelligence.ai/blogs/model-context-protocol-mcp-security-checklist/].
- **Registries.** Official `registry.modelcontextprotocol.io` ~500 entries, Glama 6000+, mcp.so 5000+, Smithery 2000+, Apigene 251 vendor-verified [Apigene: https://apigene.ai/blog/mcp-marketplace; TrueFoundry: https://www.truefoundry.com/blog/best-mcp-registries]. The official registry has **no security scanning**; trust must be enforced by the *client*.

### 3.2 LangGraph

## 6. Refined / Invented Features (10) — Fully Specified

> All ten respect: local-first, single-user workstation, trust gates, paid-call gates, no unsafe network, stdio-first MCP, days-to-weeks buildable, grounded in repo + ecosystem evidence. None proposes public MCP HTTP, cloud hosting, multi-tenancy, generic chat UI, generic RAG, marketplace, or visual workflow builder.

### F1. Capability Card Enforcement Gate

1. **Name:** Capability Card Enforcement Gate.
2. **Status after Repo Reality Pass:** **Hardening** (cards produced + signed already; **no enforcement gate** exists).
3. **One-line pitch:** Turn already-produced signed Capability Cards into a deterministic runtime gate that refuses execution when a node's declared requirements (trust, audit, HITL, paid-call) aren't satisfied.
4. **Why now:** CoSAI's 2026 baseline mandates that "tools must have single, limited purpose and not delegate security decisions to LLMs" and that signed artifacts be verified before use [Adversa-CoSAI]; MCP supply-chain stats show 41% of official-registry servers have zero auth and 43% of all servers have unsafe command paths [NetworkIntelligence-MCP]. The repo already implements the *hardest* part — typed, hashable, ECDSA-P256 + HMAC-SHA256-signed cards (`capabilities/signing.py`) — but no call site enforces them.
5. **Why ARC/SwarmGraph is uniquely positioned:** Cards are typed against the SwarmGraph IR (`capabilities/from_ir.py`), the MCP registry (`capabilities/from_mcp.py`), and adapters (`capabilities/from_adapters.py`). No other agent framework has this triad already wired to a typed IR.
6. **User story:** *As an operator running an untrusted graph, I want ARC to refuse to invoke any IR node whose required `audit_level` exceeds the run's audit configuration, or whose `trust_level=PRIVILEGED` lacks an explicit approval, with a typed `ENFORCEMENT_DENIED` event and a clear remediation message — without touching my prompts.*
7. **Technical design:**
   - **Python modules:** new `capabilities/enforcement.py` with `enforce_card(card, ctx) -> EnforcementDecision`. Call sites: `adapters/base.RuntimeAdapter.run_workflow` wrapper, `mcp/server.py` `_trusted()` flow, `swarmgraph-sdk/swarmgraph/runner.py` per-node pre-execute hook. Reuse `security/enforcement.EnforcementContext` and emit existing `DENIED` events with new `denial_reason="capability_card_violation"`.
   - **TypeScript:** add `CapabilityDenialReason` to `packages/arc-protocol-ts/src/run-events.ts` typed union; render in `packages/arc-extension/src/browser/assurance/`.
   - **CLI shape:** `arc capabilities verify --workflow <id> [--card-dir .arc/cards] --strict` (already partially exists via `cli/capabilities.py`; add `--strict` flag and exit-code 3 on denial). `arc run <id> --enforce-cards` global flag, plus default `cards.enforce: true` in profile YAML.
   - **Protocol/event schema:** extend `RunEvent` union with `CAPABILITY_CARD_DECISION { card_id, card_hash, decision, reason, correlation_id }` (additive, not breaking).
   - **UI:** Assurance tab "Cards" sub-panel listing per-node card status (signed/unsigned, valid/invalid, decision); denial modal already exists (Phase 23.3 `DenialModal`).
   - **Storage:** `.arc/cards/<entity>.json` (and `.sig`) — already in `capabilities/signing.py` write path.
8. **MCP / tooling integration:**
   - New MCP tool: `arc_capability_verify(card_id: str) -> {ok, valid, signature_status, decision_if_run}`.
   - New MCP resource: `arc://capabilities/{card_id}`.
   - Trust gate: `_trusted()` already required; additionally require `ARC_CAPABILITIES_ENFORCE=1` env or `--enforce-cards` flag for the gate to refuse (default-on but explicit `--no-enforce-cards` for legacy).
9. **Security & safety:** Fail-closed on signature error (per repo's "fail-closed on malformed sensitive records" precedent). Never silently downgrade. Refuse unsigned cards from untrusted workspaces. Card hash is bound into the audit chain (extends `AUDIT_RECORD` event payload with `card_hash`), creating cross-evidence tying execution decisions to specific signed declarations.
10. **Implementation plan:**
    - **Day 1:** `capabilities/enforcement.py` + unit tests for the decision matrix (trust/audit/HITL/paid); wire `arc capabilities verify --strict`; protocol event addition.
    - **Week 1:** Wire into `RuntimeAdapter.run_workflow` for SwarmGraph + LangGraph adapters; add denial render to Assurance tab; cross-reference audit-chain record.
    - **Week 2+:** Wire into MCP server outbound dispatch (intersects with F2), CrewAI, OpenAI Agents, AG2 adapters; add OTel span attribute `arc.capability.card_hash`.
11. **Tests:**
    - **Unit:** decision matrix (trust × audit × HITL × paid × signature), unsigned-card refusal in untrusted workspace, hash mismatch.
    - **Integration:** SwarmGraph run with missing card → denial; signed card with audit_level=`swarmgraph_hmac` runs against `sha256`-only writer → denial.
    - **UI:** Assurance Cards panel rendering, denial modal flow.
    - **Adversarial/security:** card replay attack (use stale card after key rotation); HMAC-signed card replayed with ECDSA verifier mismatch; tampered payload; symlinked card path traversal.
    - **Regression:** prior runs without cards still succeed with `--no-enforce-cards`.
12. **Success metrics:** % of adapter runs with valid signed cards ≥ 95% within first month of opt-in; zero false-positive denials in golden-trace replay; CLI exit code 3 surface stable; median verification overhead < 5 ms/node.
13. **Risks & mitigations:** *Risk:* breaks existing runs. *Mitigation:* opt-in via profile + per-run flag; deprecation horizon. *Risk:* card drift across releases. *Mitigation:* `IR_SCHEMA_VERSION` + `CARD_SCHEMA_VERSION` migration matrix already exists. *Risk:* user fatigue. *Mitigation:* nest denials in Assurance with explainers from `cli capabilities explain`.
14. **Priority score:** Impact **5**, Feasibility **5**, Differentiation **5**, Strategic fit **5**. **Total 20/20.**
15. **MVP cut:** Day-1 + Week-1 (SwarmGraph + LangGraph + Assurance UI). One CLI command, one MCP tool, one event type.
16. **Stretch:** Bind card_hash into OTel `gen_ai.*` spans and Flight Recorder index for cross-replay verification.

### F2. MCP Tool Risk Sandbox v2 — Outbound Enforcement + Injection-Pattern Re-Score

1. **Name:** MCP Tool Risk Sandbox v2.
2. **Status:** **Partial → new enforcement layer** (manifest classifier + drift exist; no outbound gate).
3. **One-line pitch:** Refuse, downgrade, or interactively approve every *outbound* MCP tool call on a deterministic risk score combining static manifest classification + per-call argument injection-pattern scanning + workspace-root scope check.
4. **Why now:** 43% of public MCP servers have unsafe command-execution paths; 36.7% SSRF; tool poisoning via metadata is a documented attack class [NetworkIntelligence-MCP; Practical-DevSecOps; Adversa-CoSAI]. CoSAI: "tools must not delegate security decisions to LLMs"; SOC Prime: "policy using full context — prompt + user + tool + parameters + target" [SOCPrime-MCP]. The repo already has `security/injection_patterns.py` (152 LOC) and `mcp/manifests.McpToolRisk` — the parts are there; the join is missing.
5. **Why ARC uniquely:** ARC already runs as a local MCP *server* (Phase 26) and ships MCP *client* helpers in `cli/mcp.py` workbench; its `security/sandbox.py` is the deepest in any open-source agent IDE (2,746 LOC, path confinement, env allowlist, Landlock probe, deny-by-default). It can act as a *trusted local proxy* between any host (Claude Code, Cursor, etc.) and any third-party MCP server.
6. **User story:** *As a developer using Claude Code through ARC, I want every `write_file` or `network_fetch` MCP call to be re-scored at call time using the manifest risk + my workspace roots + injection-pattern scan, and either auto-allow (low), prompt me (medium), or refuse (high), without me having to write OPA rules.*
7. **Technical design:**
   - **Python modules:**
     - New `mcp/enforcement.py` `assess_call(server_id, tool_name, args, ctx) -> CallDecision` combining: (a) `McpToolRisk.risk_level`, (b) `injection_patterns.scan(json.dumps(args))`, (c) `workspace.is_inside_roots(args)` for path-like fields, (d) drift status from `ManifestStore.check_drift`, (e) bound `EnforcementContext`.
     - New `mcp/proxy.py` thin stdio proxy: ARC acts as MCP client to an upstream server and *re-exposes* the tool to a downstream host with the gate inserted.
     - Reuse `security/redaction.Redactor` for arg logging.
   - **TypeScript:** `packages/arc-protocol-ts/src/run-events.ts` add `MCP_CALL_DECISION` event.
   - **CLI shape:** `arc mcp proxy --upstream <cmd> [--policy low|medium|high] [--workspace .]` — runs ARC as a stdio↔stdio proxy with the gate. `arc mcp policy explain <server>:<tool>` shows expected decision matrix per arg shape.
   - **Protocol/event schema:** `MCP_CALL_DECISION { server_id, tool_name, decision, reason, risk_signals[], correlation_id }`.
   - **UI:** New "MCP" tab in arc-extension, or a sub-panel in Assurance; live call ledger; approve/deny buttons feeding `POST /api/enforcement/retry` (already exists).
   - **Storage:** `.arc/mcp/decisions.jsonl` rolling ledger; tied to audit chain via `AUDIT_RECORD.mcp_decision_ref`.

## 10. Final Recommendation — Build F1 First

### The single highest-leverage feature to build first: **F1 — Capability Card Enforcement Gate.**

**Why F1 beats the alternatives:**

- **Highest leverage per dev-hour.** Cards, signing, IR, simulation, run-diff are *all already built* (verified in §1). The only missing piece is the runtime gate. This is the cleanest "wire-up" payoff in the entire repo.
- **Unifies the differentiation story.** SwarmGraph's pitch (consensus, audit, replay, trust) only delivers if execution is *refused* when declarations don't match. Without enforcement, signed cards are decorative — and the prior research artifact noticed this gap when it listed "Agent Capability Cards" as a *future* item even though cards were already built.
- **Ecosystem timing.** CoSAI 2026 mandates verified signed artifacts; MCP supply-chain stats put 43% of public servers at unsafe command paths; Capability Card *enforcement* is the local-first answer to that risk class [Adversa-CoSAI; NetworkIntelligence-MCP].
- **Composability.** F1 unlocks F2 (MCP outbound enforcement reads card declarations of the upstream server), F4 (linter S-rules consume cards), F8 (eval recommendations write to cards), F10 (blast-radius card = preview of F1 decisions). Building F1 first makes the rest cheaper.
- **Risk profile.** No new external dependencies; no new protocol; no new transport; no new schema (additive `RunEvent` only); fully reversible via `--no-enforce-cards`.
- **Versus F2:** F2 is also high-leverage but requires the proxy infrastructure (more moving parts; new stdio↔stdio component). F1 is purely a function added at known call sites.
- **Versus F3:** F3 is great wedge marketing but does not change runtime safety today. F1 does.
- **Versus F5/F6/F10:** UI/UX polish; lower behaviour-change impact.
- **Versus F8:** F8 is also small, but a closed eval→policy loop without F1 only changes profile defaults — the actual gate is still missing.

### What to implement first in code (Day 1, ≤ 8 hours)

1. **Create** `python/src/agent_runtime_cockpit/capabilities/enforcement.py` with:
    ```python
    from __future__ import annotations
    from dataclasses import dataclass
    from typing import Literal
    from .models import CapabilityCard, AuditLevel, TrustLevel, HitlRequirement
    from .signing import verify_card
    from ..security.enforcement import EnforcementContext

    Decision = Literal["allow", "deny", "interactive"]

    @dataclass(frozen=True)
    class EnforcementResult:
        decision: Decision
        reason: str
        card_hash: str | None
        correlation_id: str

    def enforce_card(card: CapabilityCard, ctx: EnforcementContext, *, signed_payload: bytes | None = None) -> EnforcementResult:
        cid = ctx.generate_correlation_id()
        if signed_payload is not None and not verify_card(card, signed_payload=signed_payload):
            return EnforcementResult("deny", "capability_card_signature_invalid", card.hash(), cid)
        if card.trust_level == TrustLevel.PRIVILEGED and not ctx.privilege_approved:
            return EnforcementResult("deny", "trust_level_privileged_not_approved", card.hash(), cid)
        if card.requires_paid_calls and not ctx.allow_paid:
            return EnforcementResult("deny", "paid_call_not_allowed", card.hash(), cid)
        if card.audit_level in (AuditLevel.SWARMGRAPH_HMAC, AuditLevel.FULL) and ctx.audit_mode == "sha256":
            return EnforcementResult("deny", "audit_level_below_required", card.hash(), cid)
        if card.hitl == HitlRequirement.BLOCKING and not ctx.has_hitl_gate:
            return EnforcementResult("deny", "hitl_blocking_required", card.hash(), cid)
        return EnforcementResult("allow", "ok", card.hash(), cid)
    ```
    (Replace field names with the actual ones in `capabilities/models.py` — my read confirms the enums exist.)

2. **Add** unit tests at `python/tests/capabilities/test_enforcement.py` covering the 5 deny branches + allow + signature mismatch + unsigned-from-untrusted.

3. **Extend** `packages/arc-protocol-ts/src/run-events.ts` typed union with `CapabilityCardDecisionEvent` (additive; non-breaking).

4. **Add a CLI verb** in `python/src/agent_runtime_cockpit/cli/capabilities.py`:
    ```bash
    arc capabilities verify --workflow <id> [--workspace .] [--strict]
    ```
    Exit 0 = allow; 2 = warning; 3 = denied.

5. **Land** behind feature flag (`ARC_CAPABILITIES_ENFORCE=0` by default in Day-1 PR; flip to opt-in default in Week-1 PR after `RuntimeAdapter` wiring).

6. **Document** in `docs/SECURITY.md` and a new `docs/adr/026-capability-card-enforcement.md`.

7. **Maintainer should run** (I did not):
