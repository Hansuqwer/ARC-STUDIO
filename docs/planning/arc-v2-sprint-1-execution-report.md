## Sprint-1 findings (recorded, per "finding, not silent skip")

- **F5 (new, live-verified): the global `GET /api/events/stream` does NOT carry
  registry `RunEvent`s.** It carries daemon *notifications*
  (`{event_id, event_type: "task_state_changed"|"audit_verified"|…, run_id?, payload}`),
  a lowercase-snake namespace distinct from the SCREAMING_SNAKE registry, with SSE
  `event:`/`id:` fields populated. Registry `RunEvent`s flow on the **per-run**
  stream `GET /api/runs/{run_id}/events` (AG-UI compatible, `mode=live|replay`,
  STREAM_END terminator). The review brief's §3.3.3 assumed RunEvent on the global
  stream. Handled additively: new `DaemonNotification` type; client exposes
  `stream_notifications` (global) and `stream_run_events` (per-run, RunEvent).
  **Panel impact:** the Event Stream panel's ordered/sequence semantics attach to
  the per-run stream; the global stream is Telemetry/ViewModel-class (§3.12).
- **F6: SSE `id:` decision (review §6.7):** parser retains `id:`; resume stays
  sequence-driven (daemon-owned) — the global stream's `id` is a connection-local
  counter, unsuitable for resume.
- **F7: daemon auth + trust gates fired exactly as designed** during live testing
  (`ARC_DAEMON_TOKEN` required; `PERMISSION_DENIED` envelope for untrusted
  workspace decoded cleanly by the Rust client). Deterministic-security model
  observed working end-to-end; unauthenticated mode used only via the daemon's own
  documented local-test flag.
- **F2 (from review, reconfirmed):** fixture README documents `run-record/`,
  `workflow/`, `cockpit/` dirs that do not exist; tests enumerate actual dirs.
