# v0.7-alpha Opt-In Cloud Features — Threat Model

> **Required per `docs/policy/local-first.md` "Resolution procedure for new
> ambiguous cases" point 2.** Every opt-in remote feature must document its
> threat surface before tag.
>
> **All three features default OFF.** With no env vars set, ARC makes zero
> outbound calls beyond the user's configured LLM vendor.

---

## Feature 1 — Pricing Feed (`cloud/pricing_feed.py`)

| Question | Answer |
|----------|--------|
| What data leaves the box | An HTTP GET request to the feed URL. No request body. No user data. |
| To whom | `openrouter.ai` (default) or `models.dev` (fallback), or a user-pinned URL |
| Under what auth | None — public endpoints. No API key, no session token sent. |
| Trigger | Only when `ARC_PRICING_FEED_ENABLED=1` |

**Threats & mitigations:**

- **MITM / DNS spoof injecting modified pricing** (e.g. `input_per_million: 0.0`
  to bypass budget caps): hash-pinning. Any change from the pinned SHA-256 is
  rejected and surfaced; user must run `arc pricing-feed accept-new-hash` after
  reviewing. ARC keeps the local `VENDOR_CONFIGS` table until then.
- **Feed source compromised at origin** (OpenRouter/models.dev account takeover):
  same hash-pinning gate — any change requires explicit consent. ARC cannot
  distinguish a legitimate price update from a malicious one, so it treats
  *every* change as requiring review. This is stricter than signature
  verification (which would auto-trust a validly-signed malicious update).
- **Feed source goes offline**: fail-closed to local cache, then to the
  hardcoded `VENDOR_CONFIGS` snapshot. ARC keeps working with last-known-good
  pricing indefinitely. Quiet log line only.
- **Schema drift breaks parser**: parse failure → keep local. `extra="ignore"`
  on parsed structures; missing required fields fails closed.
- **Replay of stale feed**: cache is only used when network fails; a fresh
  successful fetch always supersedes cache.

**Residual risk:** if a user blindly runs `accept-new-hash` without reviewing
the diff, they adopt whatever the feed says. ARC's job is to surface the
change, not to prevent the user from accepting it. Documented, accepted.

## Feature 2 — Budget Broker (`cloud/budget_broker.py`)

| Question | Answer |
|----------|--------|
| What data leaves the box | `{team_id, scope, amount}` JSON. **No prompts, no code, no model IDs.** |
| To whom | User-configured `ARC_BUDGET_BROKER_URL` (team's own broker) |
| Under what auth | `Authorization: Bearer <ARC_BUDGET_BROKER_TOKEN>` |
| Trigger | Only when URL + token + team_id all set |

**Threats & mitigations:**

- **Broker compromised, returns `approved: false` for everything** (denial of
  service): user can set `ARC_BUDGET_BROKER_FALLBACK=1` (default) so an
  unreachable/hostile broker falls back to the local budget cap. The local
  `BudgetEnforcer` is always the floor.
- **Broker compromised, returns `approved: true` to over-spend**: the local cap
  still applies first. Broker can only *further restrict*, never *expand*, the
  local cap. (Integration checks local first; broker is an additional gate.)
- **Broker unreachable + `fallback_to_local=False`**: fail-closed → DENY. This
  is the explicit "team budget is authoritative, no local override" posture.
- **Token exfiltration**: the bearer token is only sent to the user-configured
  URL. ARC never logs it (redacted from event payloads — `BudgetBrokerSync`
  carries scope/amount/approval booleans only, never the token).
- **Amount/scope reveals usage patterns**: the broker learns spend cadence by
  design (that's its job — shared team budget). No prompt content is exposed.

**Residual risk:** the team broker operator sees per-instance spend timing and
amounts. This is inherent to shared-budget functionality and is the user's
explicit choice to enable. No code/prompt exfiltration possible.

## Feature 3 — Observability Bridge (`cloud/observability_bridge.py`)

| Question | Answer |
|----------|--------|
| What data leaves the box | Sanitized semantic span attributes only (tokens, cost, durations, denial counts) |
| To whom | User-configured `ARC_OBSERVABILITY_BRIDGE_URL` (Langfuse/Helicone/SigNoz/custom) |
| Under what auth | Optional `ARC_OBSERVABILITY_BRIDGE_AUTH` header |
| Trigger | URL set **AND** per-session consent granted |

**This is the highest-sensitivity feature** — it exports span data, which in a
naive implementation could contain prompts/code/context.

**Threats & mitigations:**

- **Accidental prompt/code exfiltration**: `sanitize_attributes()` strips any
  key matching the forbidden allowlist (`prompt`, `messages`, `content`,
  `system_prompt`, `tool_input`, `tool_output`, `chat_history`, `code`,
  `context`, `text`) plus any key containing those substrings. Defense-in-depth:
  even if a span carries a prompt attribute, it never leaves the box. Verified
  by `test_exported_payload_sanitized` + `test_sanitize_strips_prompt_and_code`.
- **Export without user awareness**: per-session consent required by default
  (`consent_mode=per_session`). No export until consent granted. Verified by
  `test_no_export_without_consent`. `always` and `env_only` modes are explicit
  opt-ins the user sets deliberately.
- **Destination compromised**: it receives only semantic counts (tokens, cost,
  duration). No proprietary data to steal. Worst case: an attacker learns your
  token-usage and cost metrics — not your prompts or code.
- **otel-exporter-otlp not installed**: graceful no-op + one-line warning. The
  feature degrades silently rather than crashing the run.

**Residual risk:** the destination operator sees aggregate usage metrics
(token counts, costs, durations). No prompt/code/context is recoverable from
the exported payload. This is the user's explicit, consent-gated choice.

## Cross-cutting invariants (all three features)

1. **Default OFF** — verified by `test_feature_disabled_no_outbound_call`,
   `test_disabled_without_full_config`, `test_disabled_without_url`.
2. **No LLM in any decision** (CoSAI) — all three are deterministic
   network/parse/compare logic. No model calls.
3. **Fail-closed** — every error variant keeps the local-first default.
4. **Visible when active** — `/wallet` surfaces active opt-ins; status bar shows
   a `cloud:` chip (Task 4).
5. **No secrets in events** — `BudgetBrokerSync` never carries the bearer token;
   `PricingFeedRefreshed` carries a truncated hash, not the full payload.
