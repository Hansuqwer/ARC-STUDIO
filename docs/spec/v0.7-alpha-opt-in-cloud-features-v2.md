## 2. Net delta from v1 (Task 1 only)

| Aspect | v1 | v2 |
|---|---|---|
| Pricing data source | bespoke ARC-published signed JSON | community-maintained models.dev |
| Authentication | Ed25519 signature (user pins pubkey) | SHA-256 hash pinning (user accepts changes) |
| Publishing infrastructure | ARC project must run a signed JSON CDN | none (models.dev handles it) |
| Schema | ARC-defined | models.dev TOML schema (already used by opencode) |
| Failure mode (data tampering) | fail-closed on signature mismatch | fail-closed on hash mismatch; user reviews diff |
| Failure mode (upstream down) | local cache fallback | local cache fallback (same) |
| Cross-language impact | none | none (Python-only feature) |
| Acceptance review | automatic if signature valid | explicit `arc pricing-feed accept-new-hash` |
| Threat: malicious data with valid auth | covered by "out of scope" (key compromise) | covered by review-before-accept (user sees diff) |

The v2 model is **slightly stricter** in practice (any change requires explicit user accept; v1 only required signature validation), and **much simpler** to operate (ARC depends on existing infrastructure instead of running its own).

Adjustments from v1's pricing-feed threat model:

### Pricing-feed (v2 — models.dev)

| Threat | Mitigation |
|---|---|
| **NEW: models.dev disappears / goes stale** | Local cache always primary; feed augments. ARC keeps working with last accepted hash forever. Periodic warning suggests user adopt mirror or pin a different feed. |
| **NEW: models.dev schema changes break parser** | `extra="ignore"` on all parsed dataclasses; missing-required-fields fails closed per protocol-additive-only.md |

| **NEW: dependency on third-party project goodwill** | Documented in spec as "external dependency"; user can override `feed_url` to a mirror or fork; ARC could ship its own mirror if upstream becomes unreliable. |

## 5. Decision log

**Why models.dev over rolling our own:**

1. **Already used in production by opencode** — proven enough for at least one CLI agent's use case
2. **Community maintenance** — pricing freshness becomes someone else's job (with ARC as one of many consumers, exerting pull on the data quality)
3. **Single source of truth for ARC's two pricing pipelines** — dev-time `sync_from_models_dev.py` (committed to repo) and runtime opt-in feed (this Task 1) both speak the same TOML schema
4. **Zero publishing infrastructure for ARC** — no signed JSON to host, no CDN, no key rotation, no signature library

**Why hash pinning instead of trust-on-first-use:**

TOFU lets a compromised upstream poison the first user-ever fetch. Hash pinning starts pessimistic — user MUST explicitly accept the initial hash. After that, any change requires re-acceptance. Slightly more friction for legitimate updates; much higher resistance to silent corruption.

**Why `accept-new-hash` as a CLI command, not auto-accept on user keypress:**

Per honesty-over-polish.md, user actions should be explicit and recorded. A typed command logs in shell history; a one-key prompt doesn't. Reviewability matters when "what changed in pricing yesterday" becomes a billing question.
