## 4. Threat model delta (added third-party concern)

| New threat (v3) | Mitigation |
|---|---|
| OpenRouter is a commercial company; could shut down or pivot | ARC keeps local snapshot indefinitely. Fallback to models.dev via env var. User can pin a custom URL (e.g., self-mirrored OpenRouter snapshot). |
| OpenRouter could add commercial-tier-only fields in their API | ARC parses only the documented free fields; ignores rest per `extra="ignore"` policy. |
| OpenRouter prices might subtly differ from vendor-direct prices (5.5% credit fee structure) | Wallet documentation calls this out; users routing direct (not via OpenRouter) should verify their cost-calc still matches OpenRouter's table. |

All v2 threats still apply.

## 5. Decision log additions

**Why OpenRouter primary instead of pinning to models.dev (v2's choice):**

Per `docs/research/pricing-feed-sources-comparison.md`:
- 300+ models vs ~100
- Native `is_free` / `deprecation_date` / `canonical_slug` / tiered pricing
- No auth, edge-cached, mature operationally

**Why keep models.dev as fallback instead of dropping it:**

- Different organizational backing (community + opencode vs OpenRouter
  commercial); different failure mode
- Trivial to support (one CLI flag in the sync script, one env var
  in the runtime client)
- User can self-mirror models.dev easily (it's a static JSON file in a
  public GitHub repo)

**Why not switch ARC's actual request routing to OpenRouter as a hub:**

Out of scope for v0.7. ARC routes to vendor APIs directly (or via
openai_compatible.py with custom base_url). OpenRouter at runtime is
either an opt-in pricing-data source (this Task 1) or a vendor block
in openai_compatible.py (future feature). Don't conflate.
